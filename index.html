01 · Under the Hood

The Dashboard Generation Flow

User Prompt
"Create a dashboard to monitor my logs data" — the conversation starts with a natural-language request.
Agent Determines a Skill Is Needed
The agent routes the request to the dashboard skill because a dashboard need is identified.
Skill Is Read, Instructions Are Read
The skill gives the agent dashboard creation instructions, design guidance, data-discovery steps, and the requirement to use manage_dashboard.
Skill Runs: Data Check + Operations
The agent checks indices and data, follows the instructions to plan the dashboard, then defines the tool params. The important param is mostly operations — the composed dashboard sections and panels.
"operations": [ { "operation": "set_metadata", "title": "Checkout latency" }, { "operation": "add_section", "panels": [ { "query": "p95 checkout latency over 24h", "chartType": "xy" }, { "query": "error rate by region", "chartType": "metric" } ]} ]
manage_dashboard Tool Runs
The tool receives the composed operations and executes them
Inside
Visualizations are generated
generate_esql
ES|QL is produced for each panel
Lens config
Lens API schema shapes the panel config
Dashboard Attachment Created
Outside the tool's internal generation loop, the completed dashboard attachment is created and stored for the conversation.
Conversation Continues + Preview Renders
The agent keeps the conversation going and may include an attachment tag. The frontend renders that tag as defined, showing the dashboard preview so the user can edit, save, or iterate.
02 · Architecture

Agent Builder Building Blocks

🧭
Skill
Guidance & routing. Follows instructions to create a dashboard
⚙️
Tools
Execute application operations needed to create a dashboard — manage_dashboard, generate_esql, vis generation.
📎
Attachment
Stores the work product. Dashboard API state persists across conversation turns.
🖥️
Renderer (e.g. renderCanvasContent)
Presents artifacts in chat, canvas, and the Kibana Dashboard app.
✏️
User Edits
Manipulate the artifact inside or outside the agent — iterate, save, refine.
Skills provide guidance. Tools perform application operations. Attachments preserve generated artifacts.
Renderers make artifacts usable across chat, canvas, and Kibana apps.
03 · Why Dashboards?

Capability Matrix

Capability What Dashboards Exercise Relevance to Future Kibana Apps
Intent Understanding Determine whether a dashboard is the appropriate response Any agent must understand user goals before acting
Data Discovery Inspect available indices and their mappings Most Kibana features depend on knowing what data exists
Query Generation Generate valid ES|QL for every panel Central to search, alerting, transforms, and more
Visualization Generation Produce Lens configs for charts and KPIs Reusable for any visual output surface
Design Guidance Layout, sections, grid, KPI placement Creates "lovable" outputs, not just functional ones
Stateful Editing Modify panels, sections, and layouts on follow-up Multi-turn agentic workflows need durable state
Persistence Save dashboards to the Kibana app Work products must survive beyond the conversation
Rendering Display across chat, canvas, and the Dashboard app Reusable rendering patterns for all Kibana artifacts
04 · Quality

Quality & Evals

One challenge with agent systems is dependence on LLM behaviour. Evals turn subjective behaviour into measurable outcomes.
